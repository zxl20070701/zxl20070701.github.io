预防

死锁预防（Deadlock Prevention）是通过破坏死锁的四个必要条件之一来防止死锁发生的策略。与死锁检测和恢复不同，死锁预防是一种积极的策略，它通过系统设计来确保死锁永远不会发生。

死锁预防的基本思想

死锁的产生必须同时满足四个必要条件：互斥条件、占有且等待条件、不可抢占条件和循环等待条件。死锁预防的基本思想是通过破坏这四个必要条件中的至少一个，来防止死锁的发生。

破坏互斥条件

互斥条件是指资源一次只能被一个进程占用。如果能够使资源可以同时被多个进程共享，就可以破坏互斥条件，从而防止死锁的发生。

实现方法

  • 资源共享：将独占资源改造为共享资源
  • 虚拟资源：通过虚拟化技术，将一个物理资源虚拟为多个逻辑资源
  • 资源复制：为每个进程提供资源的副本

应用实例

  • 只读文件：多个进程可以同时读取同一个文件
  • 打印机缓冲：使用打印机缓冲技术，允许多个进程同时向打印机发送数据
  • 内存共享：多个进程可以共享同一块内存区域

局限性

并非所有资源都可以改造为共享资源，有些资源本质上就是独占的，如打印机、磁带机等物理设备。

破坏占有且等待条件

占有且等待条件是指一个进程已经占有了至少一个资源，但又请求新的资源。如果能够确保进程在请求资源时不会占有其他资源，就可以破坏占有且等待条件，从而防止死锁的发生。

实现方法

  • 静态分配：进程在执行前一次性申请所有需要的资源
  • 资源预分配：系统在进程创建时就为其分配所有可能需要的资源
  • 原子操作：将资源请求和释放操作设计为原子操作

应用实例 

// 静态分配示例
// 进程一次性申请所有需要的资源
request(R1, R2, R3);  // 同时请求资源R1、R2、R3
use_resources();      // 使用资源
release(R1, R2, R3);  // 同时释放资源R1、R2、R3

局限性

  • 资源利用率低：进程可能长时间占有不需要的资源
  • 进程创建困难：进程需要预先知道所有需要的资源
  • 系统开销大：需要预分配大量资源

破坏不可抢占条件

不可抢占条件是指已分配给进程的资源不能被强制性地抢占。如果能够允许系统强制性地抢占进程占有的资源，就可以破坏不可抢占条件，从而防止死锁的发生。

实现方法

  • 资源抢占：当进程请求的资源不可用时，抢占该进程已占有的资源
  • 状态保存：抢占资源时，保存进程的状态，以便后续恢复
  • 资源回收：系统可以强制性地回收进程占有的资源

应用实例 

// 资源抢占示例
// 进程A占有资源R1，请求资源R2
// 如果资源R2被进程B占有，且进程B正在等待其他资源
// 则可以抢占进程B占有的资源R2
if (resource_R2_unavailable) {
    if (process_B_waiting_for_other_resources) {
        preempt_resource_R2_from_process_B();
        allocate_resource_R2_to_process_A();
    } else {
        block_process_A();
    }
}

局限性

  • 实现复杂：需要保存和恢复进程状态
  • 系统开销大：频繁的抢占和恢复操作会增加系统开销
  • 应用限制：并非所有资源都可以被抢占

破坏循环等待条件

循环等待条件是指存在一组进程，每个进程都在等待其他进程所占有的资源，形成一个循环等待链。如果能够确保系统中不会出现循环等待，就可以破坏循环等待条件，从而防止死锁的发生。

实现方法

  • 资源有序分配：为系统中的所有资源类型分配一个全局唯一的编号，进程必须按照编号递增的顺序请求资源
  • 资源层次结构：将资源组织成层次结构，进程只能按照层次结构的顺序请求资源
  • 资源分类：将资源分成不同的类别，进程只能按照类别顺序请求资源

应用实例 

// 资源有序分配示例
// 为资源分配全局编号：R1=1, R2=2, R3=3
// 进程必须按照编号递增的顺序请求资源
request(R1);  // 请求资源R1
request(R2);  // 请求资源R2
request(R3);  // 请求资源R3
use_resources();
release(R3);  // 释放资源R3
release(R2);  // 释放资源R2
release(R1);  // 释放资源R1

局限性

  • 资源编号困难：需要为所有资源类型分配全局唯一的编号
  • 资源申请顺序受限：进程必须按照特定顺序申请资源
  • 系统灵活性降低：限制了进程对资源的申请顺序

死锁预防策略的比较

预防策略 实现难度 系统开销 资源利用率 适用性
破坏互斥条件 困难 小 高 有限
破坏占有且等待条件 简单 大 低 广泛
破坏不可抢占条件 中等 中等 中等 有限
破坏循环等待条件 简单 小 中等 广泛

死锁预防的应用实例

数据库系统

数据库系统通常采用资源有序分配策略来预防死锁。事务必须按照特定的顺序请求锁资源，从而避免循环等待。

操作系统

操作系统通常采用多种策略组合来预防死锁。例如，Linux内核使用资源有序分配策略来管理内核资源，同时使用超时机制来避免无限期等待。

分布式系统

分布式系统通常采用超时机制和事务回滚来预防死锁。如果一个事务在一定时间内无法获得所有需要的资源，系统会回滚该事务，释放已占有的资源。

死锁预防的优缺点

优点

  • 简单易实现：只需要破坏死锁的一个必要条件即可
  • 可靠性高：只要正确实现了预防策略，就可以确保死锁永远不会发生
  • 系统开销小：相比死锁检测和恢复，预防策略的系统开销较小

缺点

  • 资源利用率低：某些预防策略可能导致资源利用率降低
  • 系统灵活性差：某些预防策略限制了系统的设计和使用
  • 适用性有限：并非所有预防策略都适用于所有类型的系统和资源
