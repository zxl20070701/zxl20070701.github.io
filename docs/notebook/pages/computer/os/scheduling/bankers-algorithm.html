银行家算法

银行家算法（Banker's Algorithm）是由艾兹格·迪杰斯特拉（Edsger Dijkstra）提出的一种死锁避免算法。该算法模拟银行家向客户贷款的方式来管理资源分配，通过安全性检查来确保系统不会进入死锁状态。

银行家算法的基本思想

银行家算法的基本思想是：当进程请求资源时，系统先假设分配这些资源，然后检查系统是否仍然处于安全状态。如果处于安全状态，则真正分配资源；否则，进程必须等待。

安全状态是指系统能够按照某种顺序为每个进程分配资源，使每个进程都能顺利完成。如果系统处于安全状态，则不会发生死锁；如果系统处于不安全状态，则可能发生死锁。

银行家算法的数据结构

银行家算法需要维护以下数据结构：

可利用资源向量（Available）

一个长度为m的向量，表示系统中各类资源的可用数量。Available[j] = k表示系统中第j类资源有k个可用。

最大需求矩阵（Max）

一个n×m的矩阵，表示每个进程对各类资源的最大需求量。Max[i][j] = k表示进程i对第j类资源的最大需求量为k。

分配矩阵（Allocation）

一个n×m的矩阵，表示每个进程已经分配到的各类资源的数量。Allocation[i][j] = k表示进程i已经分配到k个第j类资源。

需求矩阵（Need）

一个n×m的矩阵，表示每个进程还需要各类资源的数量。Need[i][j] = Max[i][j] - Allocation[i][j]。

银行家算法的步骤

安全性算法

安全性算法用于检查系统是否处于安全状态：

  1. 初始化：令Work = Available，Finish[i] = false（对于所有i）
  2. 查找：查找满足以下条件的进程i：
    • Finish[i] = false
    • Need[i] ≤ Work
  3. 如果找到这样的进程，则执行以下操作：
    • Work = Work + Allocation[i]
    • Finish[i] = true
    • 转到步骤2
  4. 如果所有进程的Finish[i] = true，则系统处于安全状态；否则，系统处于不安全状态

资源请求算法

当进程Pi发出资源请求Request[i]时，系统执行以下步骤：

  1. 如果Request[i] ≤ Need[i]，转到步骤2；否则，请求出错，因为进程请求的资源超过了它声明的最大需求量
  2. 如果Request[i] ≤ Available，转到步骤3；否则，进程Pi必须等待，因为系统中没有足够的可用资源
  3. 系统尝试为进程Pi分配资源：
    • Available = Available - Request[i]
    • Allocation[i] = Allocation[i] + Request[i]
    • Need[i] = Need[i] - Request[i]
  4. 执行安全性算法，检查系统是否仍处于安全状态：
    • 如果安全，则真正分配资源，进程Pi获得资源
    • 如果不安全，则撤销分配，进程Pi等待

银行家算法的示例

假设系统中有5个进程（P0-P4）和3类资源（A、B、C），资源A、B、C的总数分别为10、5、7。

初始状态

进程 Allocation Max Need Available
P0 0 1 0 7 5 3 7 4 3 3 3 2
P1 2 0 0 3 2 2 1 2 2
P2 3 0 2 9 0 2 6 0 0
P3 2 1 1 2 2 2 0 1 1
P4 0 0 2 4 3 3 4 3 1

安全性检查

检查系统是否处于安全状态：

  1. 初始：Work = Available = (3, 3, 2)，Finish = (false, false, false, false, false)
  2. 查找满足Need[i] ≤ Work的进程：
    • P0：Need[0] = (7, 4, 3) > Work，不满足
    • P1：Need[1] = (1, 2, 2) ≤ Work，满足，选择P1
  3. 分配资源给P1：Work = Work + Allocation[1] = (3, 3, 2) + (2, 0, 0) = (5, 3, 2)，Finish[1] = true
  4. 继续查找满足条件的进程，最终可以找到安全序列：P1 → P3 → P4 → P0 → P2

资源请求示例

假设进程P1请求资源Request[1] = (1, 0, 2)：

  1. 检查Request[1] ≤ Need[1]：(1, 0, 2) ≤ (1, 2, 2)，满足
  2. 检查Request[1] ≤ Available：(1, 0, 2) ≤ (3, 3, 2)，满足
  3. 尝试分配资源：
    • Available = Available - Request[1] = (3, 3, 2) - (1, 0, 2) = (2, 3, 0)
    • Allocation[1] = Allocation[1] + Request[1] = (2, 0, 0) + (1, 0, 2) = (3, 0, 2)
    • Need[1] = Need[1] - Request[1] = (1, 2, 2) - (1, 0, 2) = (0, 2, 0)
  4. 执行安全性检查，发现系统仍处于安全状态，因此可以真正分配资源

银行家算法的优缺点

优点

  • 安全性高：可以确保系统永远不会进入死锁状态
  • 资源利用率高：相比死锁预防策略，资源利用率更高
  • 动态性：可以根据进程的请求动态地分配资源

缺点

  • 信息需求大：需要预先知道每个进程的最大资源需求量
  • 计算复杂度高：每次资源请求都需要进行安全性检查
  • 系统开销大：安全性检查需要消耗大量的CPU时间和内存空间
  • 保守性：算法过于保守，可能导致资源利用率降低

银行家算法的应用

银行家算法主要应用于以下场景：

  • 批处理系统：在批处理系统中，可以使用银行家算法来管理资源分配
  • 数据库系统：在数据库系统中，可以使用银行家算法来管理事务的资源请求
  • 分布式系统：在分布式系统中，可以使用银行家算法来协调多个节点之间的资源分配

银行家算法的改进

为了克服银行家算法的缺点，研究人员提出了一些改进算法：

  • 概率银行家算法：通过概率分析来减少安全性检查的频率
  • 分层银行家算法：将资源分成多个层次，分别进行管理
  • 分布式银行家算法：将银行家算法扩展到分布式环境中
