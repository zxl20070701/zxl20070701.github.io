死锁条件

死锁（Deadlock）是指在多进程环境中，一组进程中的每个进程都在等待其他进程所占有的资源，导致所有进程都无法继续执行的一种状态。死锁是操作系统中一个重要的并发控制问题，理解死锁的条件和产生原因对于预防和解决死锁问题至关重要。

死锁的定义

死锁是指在一组进程中，每个进程都在等待其他进程所占有的资源，而这些资源又无法被释放，导致所有进程都无法继续执行的状态。死锁状态下的进程称为死锁进程，它们相互等待，形成了一个循环等待链。

死锁的四个必要条件

死锁的产生必须同时满足以下四个必要条件，只要破坏其中一个条件，就可以预防死锁的发生。

互斥条件（Mutual Exclusion）

资源一次只能被一个进程占用，不能同时被多个进程共享。如果其他进程请求该资源，请求进程必须等待，直到占有资源的进程释放该资源。

例如，打印机是一种典型的互斥资源，一次只能被一个进程使用。如果多个进程同时请求使用打印机，只有一个进程能够获得打印机，其他进程必须等待。

占有且等待条件（Hold and Wait）

一个进程已经占有了至少一个资源，但又请求新的资源，而这些新资源又被其他进程占有，导致请求进程被阻塞，但它仍然保持着已占有的资源。

例如，进程A已经占有了资源R1，但又请求资源R2，而资源R2被进程B占有，进程A被阻塞，但它仍然保持着资源R1。

不可抢占条件（No Preemption）

已分配给进程的资源不能被强制性地抢占，只能由占有该资源的进程主动释放。

例如，如果进程A占有了资源R，其他进程不能强制性地从进程A手中夺取资源R，只能等待进程A主动释放资源R。

循环等待条件（Circular Wait）

存在一组进程{P1, P2, ..., Pn}，其中P1等待P2占有的资源，P2等待P3占有的资源，...，Pn等待P1占有的资源，形成一个循环等待链。

例如，有三个进程P1、P2、P3和三个资源R1、R2、R3，如果P1占有R1并请求R2，P2占有R2并请求R3，P3占有R3并请求R1，则形成了一个循环等待链。

死锁的例子

考虑以下场景：有两个进程P1和P2，以及两个资源R1和R2。

  1. 进程P1占有了资源R1，进程P2占有了资源R2
  2. 进程P1请求资源R2，进程P2请求资源R1
  3. 由于资源R2被进程P2占有，进程P1被阻塞
  4. 由于资源R1被进程P1占有，进程P2被阻塞
  5. 进程P1和P2相互等待对方释放资源，形成了死锁

死锁的建模

死锁可以用资源分配图（Resource Allocation Graph）来建模和描述。

资源分配图

资源分配图是一个有向图，用于描述进程和资源之间的关系。图中包含两种节点：

  • 进程节点：用圆圈表示，表示系统中的进程
  • 资源节点：用方框表示，表示系统中的资源

图中包含两种边：

  • 请求边：从进程节点指向资源节点，表示进程请求该资源
  • 分配边：从资源节点指向进程节点，表示该资源已分配给进程

死锁的判定

如果资源分配图中存在环路，且环路中的每个资源节点都只有一个实例，则系统处于死锁状态。如果资源分配图中不存在环路，或者环路中的某些资源节点有多个实例，则系统不处于死锁状态。

死锁与饥饿的区别

死锁和饥饿都是进程调度中的问题，但它们有本质的区别：

比较项 死锁 饥饿
定义 一组进程相互等待对方所占有的资源 一个进程长期得不到所需的资源
涉及进程数量 至少涉及两个进程 可以只涉及一个进程
资源状态 资源被其他进程占有 资源可能被其他进程占有，也可能被系统保留
解决方法 预防、避免、检测与恢复 公平性调度算法

死锁的影响

  • 系统资源浪费：死锁进程占有的资源无法被其他进程使用
  • 系统吞吐量下降：死锁进程无法继续执行，降低了系统吞吐量
  • 系统响应性降低：死锁进程无法响应用户请求
  • 系统稳定性降低：死锁可能导致系统崩溃或需要人工干预

死锁的检测

检测死锁的方法主要有：

  • 资源分配图法：通过检测资源分配图中是否存在环路来判断是否发生死锁
  • 银行家算法：通过模拟资源分配过程来检测是否会发生死锁
  • 超时法：如果一个进程等待资源的时间超过预设的阈值，则认为可能发生死锁
